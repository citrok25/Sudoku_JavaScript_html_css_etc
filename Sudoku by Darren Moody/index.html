I am creating a Game project / Web App using a Grid and

will be developing my own version of the game Sudoku!

This version will use the MVC design pattern and will contain

a Model, View, and Controller.

  • The Model contains functions, parameters and information about the state of the game, .
    • The 'game pieces' will be numbers contained within the cells of a table.
    • The information such as a piece's location and value is manipulated, or changed, by the
      user's controls, or the Controller, as laid out in the Viewside of the MVC pattern.
    • The Controller will be contained within the View.
  • The View instantiates the interface for the game-play, how the game is displayed
    to the user, including what the gameboard and game objects will look like.
    • This View is primarily where the look and feel of the game is produced.
    • In a nutshell, the view produces the game interface, or game-board design.
  • The Controller.
    • "For this project, I'm not going to worry about separating the controller from the view into
      a separate source file."
    • For the case of Sudoku, I will have interactive cell objects and buttons, a 'helper' keypad
      that gives the player the possible values possible for a particular cell, and a drop-down menu.
    • All of these interactive objects would be considered the Controller side of the MVC pattern.
    • Each object will contain parameters as needed for the gameplay.
    • The Model for this game will contain and track the information for all of these things.

Playing this Version of Sudoku:

  • My Sudoku game will be generating a Sudoku table that is made up of 81 separate cell objects
    in a 9 x 9 table. The 9 x 9 table will be made up of 9 separate 3 x 3 tables, which I will call mini-grids.
    My functions for generating the table will be passing objects as parameters, which objects will
    contain the parameters for creating the table size, as well as states contained in arrays.
    The parameters contained in the objects will be shown below.
  • The player will see a blank game-grid and other elements of the game's interface
  • The player must select a difficulty from the drop-down list of difficulties and a new puzzle of that difficulty will auto-load on selection.
  • Each new puzzle generated will be UNIQUE and RANDOM, having been created using a back-tracking algorithm
  • A certain number of cells will be left blank on the game-grid depending on the difficulty chosen.
    The harder the difficulty chosen, the more blank cells you will start out with.
  • The player can select any empty-cells on the game-grid and the possible correct answers for that cell will be given in a separate clickable grid called the keypad.
  • The player can click a 'Check Answers' button to check their answers at any time during the game.
  • If all the answers are correct a message will say "everything is good so far", or if there is a conflict the conflicting cells will be highlighted in red".
  • Once all of the numbers have been filled in and all of the answers are correct, you have won the game! The game is over and a 'you win' message will display.

Winning the game:

  • In order to win at Sudoku you must enter the numbers 1 through 9 in each row, column, and mini-grid (3 x 3 table
    within the larger table grid). You may enter the numbers in any order but you cannot repeat the same number twice
    in any corresponding row, column, or mini-grid. For example, if there is already a 6 in a given row, then there
    must not be another 6 in the same horizontal row (across the entire board), vertical column (across the entire board)
    or in the same 3 x 3 mini-grid, or else one of the sixes must be in the wrong place...this is called a "conflict"!!!
    Once you have filled in the entire Sudoku Game Board with numbers 1 - 9, having no conflicts, YOU WIN!
